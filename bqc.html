Every capability Bubble runs today
maps to Go + Postgres.

I walked all six current use cases and every integration through the proposed stack. It holds. The one new operational piece is a Postgres-backed job runner for the nightly syncs.
Rather than just pick a stack, I traced every capability the PDF says Bubble does today and checked it fits. Short version: it all lives in Go + Postgres, and the only new infra is a background job runner, which is still just Postgres. Let me walk the use cases, then the integrations, then the one gap.
Core use cases · 1 of 2

Live forecast, approvals, scenarios

Daily live forecast
HRIS-driven forecast
Scheduled job pulls Personio / HiBob / Bamboo in parallel, normalises each to our model, idempotent upsert to Postgres.
Lever: a unified HRIS API (Kombo, EU) collapses 3 integrations into 1.
Complexity high+ scheduler
Approval management
Configurable flows
Config-driven rule table (JSONB conditions) + a state machine in Go. Route by cost center / role / amount; notify via SendGrid + Slack.
No Temporal needed at this volume: approvals are human-speed.
Complexity modStack fits
Budgeting & scenarios
Base / freeze / growth
scenario_id on the fact table. Clone = copy rows; compare = query two side by side. Rollups via materialized views.
Small data: Postgres aggregates tens of thousands of rows in ms.
Complexity modStack fits
The forecast is the hard one, and it's hard because of external APIs, not our side. Approvals are a state machine in Postgres; scenarios are just rows the database aggregates in milliseconds. The data is small enough that both stay simple.

Deviation, upload, access control

Deviation analysis
Plan vs actuals
A SQL join + aggregation of plan (scenario) against actuals (HRIS / uploads), grouped by cost center and month. Materialized view for the read.
Migrates first: this is the read-heavy Month-1 workload.
Complexity lowStack fits
Single source of truth
CSV / Excel upload
Endpoint → parse (encoding/csv, excelize) → validate → column-map → idempotent upsert. Raw files in object storage.
Real work: validation UX and column mapping, not parsing.
Complexity mod+ object store
Data privacy & access
Admin vs manager scope
Postgres Row-Level Security for tenant + role scoping. Manager sees their org subtree via recursive CTE / ltree.
Selling point: Bubble privacy rules → RLS is a clean 1:1. API + MCP inherit it.
Complexity mod-highStack fits
Deviation is almost free in SQL. Upload's difficulty is telling a user which rows failed and why, not reading the file. Access control is the one I'd test hardest, because a financial-data leak is catastrophic, but it maps beautifully: their privacy rules become Postgres RLS, and everything downstream inherits it.

The one new piece

Where the scheduled jobs run

The honest gap
Postgres-backed job runner: River

HRIS / ATS syncs, materialized-view refreshes, and approval reminders all need scheduled background work. In a single-binary deployment, that's the one piece we hadn't named yet.

  • Runs inside the same Go binary. No extra service to operate.
  • Uses the Postgres we already have. Durable jobs, retries, periodic schedules, no new datastore.
  • Survives restarts. A crashed nightly sync resumes; nothing silently dropped.
It ships and scales with the API, so scheduled work stays as easy to operate as everything else.
The natural question a technical founder asks: where do the nightly jobs actually run? A Postgres-backed queue lives in the same binary and uses the same database, so it's durable and observable without adding a moving part to on-call.
Verdict

Every capability fits, end to end

Addition 1
Job scheduler

River, Postgres-backed, in the binary. For the nightly syncs.

Addition 2
Object storage

EU bucket / Supabase Storage. For raw CSV / Excel uploads.

Addition 3
Org hierarchy

Recursive CTE / ltree. A schema detail, not infra, for manager scoping.

Complexity, hardest first: HRIS / ATS integrations → multi-tenant RLS + manager subtree → recalc / scenarios → approvals + config migration → CSV import → deviation → the keep-as-is integrations.
The line: I didn't just pick a stack, I walked every current capability through it. It all fits in Go + Postgres, with three small additions.
The whole exercise is really the proof: the stack isn't a guess, it's what's left after checking every box. Three small additions, and nothing that a solo engineer can't operate day to day.
