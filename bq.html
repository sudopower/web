Basqo · migration case · Kiran Nambiar

Migrate the backend first.
Keep Bubble as the UI.

The pain is Bubble's recalculation ceiling and lack of code control: a backend problem, not scale. So I'd fix that with the smallest change one engineer can own.
~30 customer companies, 200 MAU, 30-40 DAU, so the challenge is getting the backend right, not handling load. I'll cover the bottleneck, what I'd ask first, the stack, the alternatives, the migration, and dates. The headline decision: backend-first, Bubble stays the frontend.

Why Bubble hits a ceiling
  • Interpreted, slow per op. Summing thousands of rows in visual workflows takes seconds-to-minutes, not milliseconds.
  • Metered compute. Every recalculation burns paid Workload Units, so it's slow and costly.
  • No code-level control. Can't recompute only what changed, cache, precompute rollups, or parallelise scenarios. The levers don't exist.
I'm not an FP&A expert, I'll lean on you for domain. But the engineering shape is clear: the core operation is a cascading aggregation, and that's exactly what a database and native code do orders of magnitude faster than a no-code interpreter that also meters you per op. The bottleneck is compute they can't optimise, not traffic. That's what Postgres + Go fixes directly.
02 · What I'd want to know first

Four questions that shape everything

  • Which workflows hit the recalculation ceiling? Scenario rollups, deviation, approvals. That's the hard part to get right.
  • Which of the ~70 types are core vs disposable? Confirms the ~8-10 entities I'd actually model.
  • Bubble plan + Data API / API Connector limits? Drives how I design the endpoints and the sync.
  • EU residency: are sub-processors DPA'd, can calls run client-side? It's a hard requirement, so it shapes the architecture.
I didn't pretend to know the internals. These are the four answers I'd want before committing. Notice they're about the domain and the constraints, not the tech.

Several stacks would work at this scale; the deciding variable is who's building it. The heavy-infra options buy scale we don't have yet and cost operability a solo engineer can't spare, so they're a deliberate no for now, not forever. I chose for fluency and operability, not fashion. If recalc ever gets heavily numeric, a small Python service is the escape hatch.
05 · The migration

Two-phase strangler: dual-run, per-customer cutover

P1 · model
~70 types → ~8-10 entities · AI-assisted from /meta, verified by hand
P1 · sync
Backfill + nightly delta into Postgres · reconciliation harness diffs vs Bubble
P1 · strangle
Repoint Bubble via API Connector · reads first, then writes + heavy logic into Go
P1 · cut over
Customer-by-customer · gated by a clean diff, reversible, Bubble as fallback
P2 · later
Swap Bubble UI for a custom frontend · same API, off the critical path
Zero regression doesn't come from a perfect script; it comes from running both systems, diffing them continuously, and flipping one customer at a time with a rollback.

07 · Questions I anticipated

Discussion prep

API Connector handoff
Have you used it? How does the deviation view actually migrate?
Map response params, build AI-generated Postman test suite covering all filters. Switch connector URL once new endpoint passes every scenario. UI sees no change.
Write conflicts
Two systems, bursty writes. Who wins?
Conflict doesn't arise if sequenced correctly. Phase 1 = reads only; Bubble stays write source of truth. Per-customer window: pause writes, reconciliation green, then flip. No dual-write.
HRIS sync timing
You pick 3AM, but the nightly HRIS pull also runs at night.
Harness tracks nightly sync completion as a hard precondition. Cutover never starts until all sync tasks done AND reconciliation is green. Need full workflow schedule first.
Solo engineer risk
What if you're sick for two weeks mid-migration?
Runbooks + observable systems, not personal promises. If 30 customers share one connector: one cutover, not 30. Dashboard anyone can read; rollback = one command.
Why Go
We can't read Go. Why not TypeScript?
Fluency = fastest debug at 2AM. Single binary = trivial solo on-call. Explicit error handling = can't silently swallow errors on financial data. Goroutines for scenario rollup concurrency.
Migration timeline
When does the first customer see the new backend? When are we off Bubble?
Month 1: first read cutover. Month 3: first write feature. Month 6: all workflows. Month 8: Bubble cost audit. Two-week app audit sharpens all numbers.
New features mid-migration
Customer needs a feature during the window. Build in Bubble or new system?
Build in Go, never in Bubble backend. New features minimised by agreement. First 2 weeks I get comfortable wiring Bubble connectors myself, so founders aren't the bottleneck.
Approval workflow migration
Customer-specific routing logic. How do you move it?
Build a config-driven rule table (straightforward). Migrate configs via Bubble app-definition export + parser, or agentic extraction with Claude. Manual admin UI as fallback.
These came up in prep. Happy to go deeper on any of them.
